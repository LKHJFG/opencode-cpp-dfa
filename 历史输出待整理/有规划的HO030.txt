C++ SEMANTIC-LEVEL VARIABLE TRACING - HANDOFF
==============================================

目标概述
--------
在 OpenCode 的 static-analysis-plugin 中实现 C++ 数据流分析（DFA）能力：
给定 C++ 代码的变量及位置，能够基于数据流分析实现语义级变量追踪
（前向追踪变量的流向、后向追踪变量的来源）。

当前版本：v0.3.0（已构建通过，但 v3 跨函数代码已合入但未打新版本 tag）
实际代码包含三个层级，按优先级降序：

  v3（最新）: AST-based 跨函数 DFA
    文件: cpp-parser.ts (~490 行) + ast-to-cfg.ts (~600 行)
          + cross-function-dfa.ts (~400 行)
    方法: 通过 web-tree-sitter WASM 解析 AST → 构建每个函数的独立 CFG
          → 构建函数调用图（Call Graph）→ 跨函数边界追踪变量
    入口: variable-trace.ts 的 execute() 优先尝试 v3，失败下降至 v2/v1
    状态: 已实现，在 variable-trace.ts 中作为首选路径
    注意: 当前无专项测试覆盖 v3（v3 只能通过集成测试间接验证，
          目前的 cpp-analysis.test.ts 测试覆盖的是 v1 路径）

  v2（中间层）: AST-based 单函数 CFG
    文件: ast-to-cfg.ts (17,952 行)
    方法: 走 tree-sitter AST 但只处理单个函数，不做过程间分析
    状态: 在 variable-trace 中作为 v3 失败后的回退
    注意: ast-to-cfg.ts 产出与 line-scan 相同的 ControlFlowGraph 类型，
          DFA 引擎无需改动即可复用

  v1（兜底）: Line-scan based CFG + DFA（原版生产代码）
    文件: cpp-cfg.ts (14,620 行) + cpp-dataflow.ts (15,641 行)
    方法: 逐行正则匹配 C++ 语句，构建控制流图（CFG），
          然后基于 def-use chain 做数据流分析
    优点: 零外部依赖，启动快，稳定可靠，有完整 19 tests 覆盖
    缺点: 语法覆盖约 85%，复杂结构不够精准，无跨函数分析

核心管线（三级回退）
------------------------
  main()
    |
    v
  trace_variable (variable-trace.ts: ~280 行)
    |-- 接收: filePath, variableName, [line], [direction]
    |-- 输出: FlowEdge[] + allVariables[] + summary text
    |
    |-- 执行流程（三级 fallback）:
    |
    |    try v3: 跨函数 DFA（首选）
    |    ├── CppParser.getInstance().parseContent(content, absPath)
    |    ├── buildFunctionCFGs(tree, sourceLines)  → Map<funcName, FunctionCFG>
    |    ├── buildCallGraph(tree, funcCfgs)         → CallSite[]
    |    ├── findFunctionForVariable(funcCfgs, varName, line)  → 定位起始函数
    |    └── traceInterprocedural(funcCfgs, callSites, varName, func, dir)
    |            │
    |            │  跨函数参数传递:  caller arg → callee param
    |            │                   callee return → caller capture
    |            │  plus: 单函数内 DFA 模式全部保留
    |            └── 结果: edges[] + allVariables[]
    |
    |    ↓ v3 失败 / WASM 未初始化 / 解析异常
    |
    |    try v2: AST 单函数
    |    ├── CppParser + ast-to-cfg.ts
    |    ├── buildASTCFG(tree, sourceLines)  → ControlFlowGraph
    |    └── analyzeDataFlow(cfg, duInfo, ...)
    |
    |    ↓ v2 也失败
    |
    |    try v1: Line-scan 兜底（原版）
    |    ├── buildCFG(sourceLines)           → ControlFlowGraph
    |    ├── buildDefUseChains(cfg, filePath) → DefUseInfo
    |    └── analyzeDataFlow(defUse, ...)     → TraceResult
    |
    |-- 结果格式化: summary + edges + variables 列表
    |
    |
  cpp-cfg.ts (line-scan, v1 专用)
    |-- buildCFG(sourceLines: string[], functionName?: string)
    |      逐行扫描，识别:
    |        - variable_declaration（正则 DECL_PATTERN）
    |        - assignment（正则 ASSIGN_PATTERN）
    |        - function_call（正则 FUNC_CALL_PATTERN）
    |        - control flow（if/for/while/switch/return）
    |      构建 BasicBlock[]:
    |        - 每个 block 含 statements（defVars + useVars）
    |        - block 间有 successors/predecessors 边
    |
    |
  ast-to-cfg.ts (AST→CFG, v2/v3 共用)
    |-- 接收 tree-sitter AST + sourceLines
    |-- 输出 ControlFlowGraph（与 cpp-cfg.ts 相同类型）
    |-- DFA 引擎无需改动即可复用
    |-- 精确识别: 函数定义块、参数列表、if/for/while/return 边界
    |-- 比 line-scan 更准但依赖 WASM（首次调用需 ~300-800ms 加载）
    |
    |
  cpp-dataflow.ts (所有版本共用)
    |-- buildDefUseChains(cfg, filePath)
    |      扫描所有 block 的 statements,
    |      构建 definitions map（variable -> DefRecord[]）
    |      和 uses map（variable -> UseRecord[]）
    |
    |-- analyzeDataFlow(defUse, startVar, startLine, direction, filePath)
    |      核心算法:
    |        - forward: 从 startVar 出发，遍历 def-use chain，
    |                   找到所有被 startVar 直接或间接赋值的变量
    |        - backward: 从 startVar 出发，逆 def-use chain，
    |                    找到所有流入 startVar 的源变量
    |        - both: 同时前向和后向追踪
    |      边类型: assignment, composition, extraction, pointer,
    |              offset, parameter, return, shared_memory
    |
    |
  cross-function-dfa.ts (v3 核心)
    |-- buildFunctionCFGs(tree, sourceLines, filePath)
    |      遍历 AST，为每个 function_definition 节点生成独立 CFG
    |      + 提取参数名（params）、函数体范围
    |
    |-- buildCallGraph(tree, funcCfgs)
    |      解析 AST 中的 call_expression，
    |      识别 caller → callee 关系，
    |      提取实际参数变量（argVars）、返回接收变量（returnCapture）
    |
    |-- traceInterprocedural(funcCfgs, callSites, varName, startFunc, dir, filePath)
    |      Step 1: 在起始函数内前向追踪直到遇到调用点
    |      Step 2: 在 callee 函数入口匹配 argVars → params
    |      Step 3: 在 callee 内继续追踪
    |      Step 4: 如果遇到 return_statement，将返回值回传 caller
    |      Step 5: 递归处理链式调用
    |      限制: 当前深度最大 3 层（防止递归无限循环）
    |
    |
  cpp-parser.ts (AST 解析，v2/v3 共用)
    |-- 基于 web-tree-sitter WASM（0.26.8） + tree-sitter-cpp（0.23.4）
    |-- CppParser 单例类，lazy init，纯 WASM 运行（无需原生编译）
    |-- 功能:
    |      parseAST(sourceCode) -> ASTNodeInfo[]
    |      extractFunctions(sourceCode) -> FunctionInfo[]
    |      extractVariables(sourceCode) -> VariableInfo[]
    |      extractAssignments(sourceCode) -> AssignmentInfo[]
    |      extractFunctionCalls(sourceCode) -> FunctionCallInfo[]
    |      extractClasses(sourceCode) -> ClassInfo[]
    |

工具注册
--------
trace_variable 已注册为第 8 个工具（src/index.ts 第 9 行 import），
通过 createVariableTraceTool() 工厂函数注册。

描述:
  "Trace a variable through C++ source code using data flow analysis.
   Given a variable name and position, finds all variables this variable
   flows to (forward) or all variables that flow into it (backward).
   Handles basic types, pointers, structs, arrays."

参数:
  - filePath (string, required): C++ 源文件路径
  - variableName (string, required): 要追踪的变量名
  - line (number, optional): 变量定义行号（未提供则自动检测）
  - direction ("forward"|"backward"|"both", optional): 追踪方向（默认 both）

已知问题与限制
--------------
1. 行扫描精度不足
   - CFG 构建基于正则匹配，无法解析模板、宏、复杂表达式
   - 正则 DECL_PATTERN 在有多个变量声明的行可能误匹配
   - 未来方案: 集成 tree-sitter AST 做语句级精确解析

2. 指针/引用追踪为启发式
   - 识别 *p 和 &x 模式但未做别名分析
   - 同一指针指向多个变量的情况无法处理
   - 二级指针 int** pp 未充分支持
   - 智能指针 unique_ptr/shared_ptr 未特殊处理

3. 缺乏过程间分析（Interprocedural）
   - 当前仅做单函数内的 DFA
   - 识别函数调用但未进入被调用函数体追踪参数-返回
   - 跨函数追踪范围限于: 识别参数传递、识别 return 赋值
   - 无调用图（Call Graph）构建

4. 结构体位域/类型信息未解析
   - 识别 struct obj.field 模式但不解析 struct 定义
   - 无类型系统: 无法区分 int* 和 char* 的赋值语义差异
   - 数组索引 a[i] 作为变量整体追踪，而非元素级别
   - union 和位域未特殊处理

5. CFG 对复杂控制流支持有限
   - if/else 嵌套深度超过 3 层可能导致 block 划分不准
   - switch/case 基本支持但 fall-through 未处理
   - do-while 循环支持但 break/continue 的 CFG 边未标注
   - try/catch/throw 完全不支持
   - goto 完全不支持

6. web-tree-sitter 初启延迟
   - WASM 初始化首次约 300-800ms（取决于 WASM 文件 I/O）
   - CppParser 为单例，首次调用后后续无延迟
   - v3/v2 路径依赖 WASM，首次 trace_variable 调用有可见延迟
   - WASM 文件路径在 Windows 下可能因权限问题拒绝访问
     （CppParser 通过 node_modules/ 相对路径加载）

7. v3 跨函数 DFA 无专项测试
   - cross-function-dfa.ts 和 ast-to-cfg.ts 均无单元测试
   - 只能通过 variable-trace.ts 集成测试间接验证
   - 当前的 cpp-analysis.test.ts 的 19 个测试覆盖的仅是 v1 line-scan 路径
   - v3 的回退逻辑在 try-catch 中，如果 v3 无声失败则悄悄走 v2/v1，可能隐藏 bug

8. 测试覆盖缺口
   - 无 C++ 标准库 STL 容器相关测试（vector, map, string 等）
   - 无多文件场景测试（当前只分析单文件）
   - 无大文件性能基准测试
   - 无恶意输入（超大文件、无限循环代码）的稳定性测试

关键技术决策
------------
- 双轨策略: line-scan 快速交付 + AST 长期提升，不相互阻塞
- WASM 模式: 无原生编译，避免 Windows 编译工具链依赖
- 纯 TypeScript: 与 OpenCode 生态一致，无混合语言
- bun build: 非 tsc（bun 的 tree-shaking 更优，产物更小）
- 无 @ts-ignore 或 as any 抑制 TypeScript 类型错误
- 工具接口匹配 OpenCode tool() 模式，便于插件热插拔
